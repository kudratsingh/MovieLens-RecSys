# ADR 0007 — Auth Provider: Keycloak (self-hosted OIDC)

**Status:** Accepted
**Date:** 2026-07-03

## Context

Phase 3 introduces real auth on every API endpoint except `/healthz` (non-negotiable #10). Every authenticated request must resolve to a tenant, and the tenant must be trustworthy — token-carried, not client-declared — because [ADR-pending] multi-tenancy isolation will use it to select champion models, Redis key prefixes, and rate limits. Cross-tenant data leakage is the highest-severity bug class (non-negotiable #9), and the tenant-resolution surface is where that class of bug lives.

This is the *first* Phase 3 ADR by design. CLAUDE.md's recommended order pins auth first because the multi-tenancy isolation ADR that follows depends on how tenants are named and how the tenant claim reaches the middleware. Getting auth wrong ripples into every downstream Phase 3 decision — feature-store key prefixes, audit-log schema, A/B routing per Phase 6, and the synthetic-user harness that has to programmatically mint tokens for CI.

The choice is between three families:

1. **Auth0 (SaaS OAuth2/OIDC).** Zero ops, first-class multi-tenancy via Organizations, offline-dev blocked (needs internet to the Auth0 tenant even for local development).
2. **Keycloak (self-hosted OIDC).** Standard OIDC patterns, real IdP running in `docker-compose`, offline-friendly, adds a JVM service and a second Postgres to the local stack.
3. **Postgres-backed JWT (custom).** Zero new infra, reuses the existing Postgres, trivial to mint tokens in tests, no standard OIDC patterns to defend, extension to social/SSO later is code-per-provider.

The criteria that decide the choice:

- **(a) Tenant-claim story** — how cleanly the provider's identity model maps onto the next ADR's tenant-isolation decision.
- **(b) Local-dev ergonomics** — CLAUDE.md is local-first ("orchestration runtime: local first, k3s/kind optional after Phase 4"); the auth provider cannot be the reason `make up-dev` fails on an airplane.
- **(c) Phase 6 A/B routing reusability** — champion/challenger routing reads a tenant claim off every request; all three families satisfy this equally, so it's not a differentiator.
- **(d) Programmatic token minting for the synthetic-user harness and CI** — non-negotiable #11's synthetic-load smoke test has to mint valid tokens without human intervention on every serving PR.
- **(e) Design-review defensibility** — CLAUDE.md's whole point is that I can defend every architectural choice at senior level. What signal does each family send?

## Decision

The auth provider is **Keycloak, self-hosted via docker-compose**, with the following shape:

- **Realm-per-tenant.** Each tenant is a distinct Keycloak realm. Realms are the strongest isolation boundary Keycloak offers — separate user store, client store, key material, and admin surface per realm. This maps 1:1 to CLAUDE.md's "a tenant is a logical isolation boundary" and constrains the [ADR-pending] multi-tenancy isolation decision to *how the app enforces isolation given a resolved tenant*, not *how tenants are named*.
- **OIDC discovery.** FastAPI reads the OIDC discovery document at each realm's `/.well-known/openid-configuration` and caches the JWKS with a 5-minute TTL. Signature verification is against JWKS, not a shared secret.
- **Tenant resolution via issuer, not claim.** The middleware extracts `tenant_id` from the **issuer** of the token (`iss` claim, which for Keycloak is `<host>/realms/<realm-name>`). A client cannot self-declare a tenant by editing a claim — the tenant is derived from *which realm's private key signed the token*.
- **Auth flows.** Authorization Code with PKCE for the Next.js frontend; direct password grant (or client credentials) for the synthetic-user harness and CI; client credentials for service-to-service calls.
- **Local dev.** Keycloak runs as a `docker-compose` service (`quay.io/keycloak/keycloak:latest`) with a dedicated `keycloak-postgres` service. Realms are seeded from `infra/keycloak/realms/*.json` at container startup — one realm per default tenant plus a `demo` realm for the personas harness.
- **Dev-only bypass.** `Settings.dev_auth_bypass: bool = False` short-circuits the middleware when set. A runtime assertion in `Settings` raises if `dev_auth_bypass=True` and `environment != "dev"` — the check runs at import time so a production build cannot boot with the bypass on.
- **Audit log integration.** The middleware writes one row per authenticated request to the audit table (`tenant_id`, `user_id`, `endpoint`, `model_version`, `latency_ms`, `outcome`) — the table schema comes with the Phase 3 audit-log PR, not this one.

## Rationale

1. **Offline dev is a hard constraint, not a preference.** CLAUDE.md pins "orchestration runtime: local first" and "run the stack via docker-compose" throughout — the value of the local loop is that it works without cloud dependencies. Auth0 fails this test at the first tunnel: a dev tenant on Auth0 still requires internet to hit `<tenant>.auth0.com` for token issuance and JWKS fetch. This is not a "usually fine" caveat — it turns "airport dev" and "corporate-network dev" into blocked states, and it means the CI job that runs the synthetic-load smoke test now has an outbound-internet dependency. Keycloak and Postgres-JWT both clear this bar; Auth0 does not.

2. **Standard OIDC patterns send the right signal in a design review.** CLAUDE.md is explicit — "I should be able to defend every architectural choice in a senior-level design review, debug any layer when it breaks, and articulate the tradeoffs vs. alternatives — *and* I should be able to hand the running system to an enterprise SRE without needing to apologize for what's missing." An SRE reading Postgres-JWT will (correctly) ask why we didn't use OIDC and JWKS-based validation, and the answer is thin: we didn't want a container. Keycloak *is* the container answer — a real OIDC IdP with discovery, JWKS rotation, PKCE, and realm-scoped keys, running in docker-compose. This is the pattern real enterprises deploy behind Auth0/Okta/Ping, and demonstrating it end-to-end is closer to the enterprise-grade framing than either of the alternatives.

3. **Realm-per-tenant makes the next ADR smaller.** The isolation-mechanism ADR — Postgres schema-per-tenant vs. row-level security vs. FastAPI-instance-per-tenant — hinges on how tenants are named and resolved. Realm-per-tenant gives that ADR a hard input: (i) the tenant identifier is the realm slug (URL-safe, human-readable), (ii) the realm's issuer URL is the ground truth for tenant identity (forge-resistant unless Keycloak's private key is stolen), and (iii) new tenants are provisioned by importing a realm JSON, which is a scriptable operation. Attribute-mapped multi-tenancy in a single realm — where every user carries a `tenant_id` attribute — would push all of that surface into application code and would give the isolation-mechanism ADR a weaker foundation to reason from.

4. **Programmatic token minting is a first-class requirement, and Keycloak handles it cleanly.** Non-negotiable #11 requires a synthetic-load smoke test on every PR that touches `src/serving/`. That test has to boot the auth provider, mint tokens for synthetic tenants and users, and exercise the API — with no human in the loop. Keycloak supports two paths: the admin API for provisioning users and roles per test run, and the direct password grant (or client credentials) for minting tokens once users exist. Both are stable OIDC surfaces documented for the last decade. The synthetic-user harness will keep a `synthetic/auth/` helper that hides the flow — the tests just ask for "a token for tenant X, user Y." Auth0 supports the same via its Management API but requires M2M app credentials committed to CI's secret store *and* internet from the CI runner. Postgres-JWT is trivial (call a Python function) but skips the whole "token was issued by a real IdP" property that the smoke test is meant to verify.

5. **JVM startup cost is real but bounded.** Keycloak's Quarkus distribution starts in 8–15 seconds cold on a modern laptop and stays running across every dev-loop iteration — you pay the cost once per session, not once per request. In CI, the auth container is up alongside Postgres and MLflow already; the marginal cost is ~15 seconds on the synthetic-load smoke test's setup, which is well under the 5-minute budget non-negotiable #11 implies. Optimizations like pre-built native images (Keycloak Quarkus supports `kc.sh build --optimized`) are available if the startup ever becomes the tight constraint, but shipping the default JVM image first is the honest baseline.

6. **The choice we can revisit cheaply if we're wrong.** Keycloak-in-front-of-FastAPI is an integration boundary — the middleware validates a Bearer token, resolves an issuer to a tenant, and moves on. If we discover in Phase 4 or Phase 5 that Auth0 or a Postgres-JWT flow would have been better, the swap is contained: reissue tokens from the new source, point the middleware at the new JWKS (or the new HMAC key), and the rest of the system does not care. Realm-per-tenant would need a data-migration step, but the *middleware contract* — "resolve a Bearer token to a `(tenant_id, user_id)` pair" — is stable across providers. This is what makes Keycloak the right starting point rather than the permanent answer.

## Alternatives considered

- **Auth0 (SaaS OAuth2/OIDC).** Considered seriously — enterprise-standard, zero-ops, Organizations feature is a genuinely good multi-tenancy primitive. Rejected on rationale #1: it fails the offline-dev requirement and drags an outbound-internet dependency into CI. Also has weaker portfolio depth ("configured Auth0 via dashboard") and vendor lock-in that this project has no reason to accept given the multi-tenancy story fits equally well in Keycloak realms. If the eventual production home for this system is a cloud where Auth0 is already the standard, the middleware contract in rationale #6 is what makes the migration bounded.

- **Postgres-backed JWT (custom).** Considered seriously — smallest surface, no new infra, trivial token minting in tests. Rejected on two grounds. First, it skips the whole standard-OIDC-patterns question that a senior reviewer asks — no discovery, no JWKS, no PKCE, no rotation story out of the box, all of which we'd end up building or bolting on. Second, "roll your own auth" is a bug-magnet category even when the surface is small: password hashing, rate limiting on `/login`, revocation, replay defense, and signing-key rotation are all things to get right by hand where Keycloak gets them right by default. The one property Postgres-JWT wins on outright — trivial token minting in tests — is achievable in Keycloak via the direct password grant with ~10 lines of helper code.

- **Attribute-mapped multi-tenancy inside a single Keycloak realm.** Considered as the "one realm, many tenants" alternative to realm-per-tenant. Rejected on rationale #3: it pushes the tenant-resolution surface into application code (read the `tenant_id` attribute claim on every request) rather than deriving tenant from the issuer URL. That's a weaker foundation for the next ADR — a bug in the attribute-mapper on a single client can leak a tenant claim into the wrong token, and the middleware has no independent check because the issuer is the same for all tenants. Realm-per-tenant makes that class of bug require Keycloak-level compromise, not application-level.

- **Google/GitHub OAuth (social sign-on only).** Considered as a lightweight, portfolio-friendly option — the frontend authenticates users via Google, the backend trusts Google-issued JWTs. Rejected because it has no multi-tenancy story that fits CLAUDE.md's model. Multi-tenancy would have to be layered on top by mapping Google identities to app-level tenant records, which lands us back in Postgres-JWT territory with an extra hop for a token we're not the issuer of. Social federation *into* Keycloak is a strict superset of this option and available if we want it later.

- **Skip auth entirely for Phase 3 and add it in Phase 5 or 6.** Not seriously considered but worth naming: this is what the pre-2026-06-02 scope did, and CLAUDE.md's scope shift is the reason it's off the table. Non-negotiable #10 pins the requirement.

## Consequences

- **docker-compose additions.** `keycloak` service (`quay.io/keycloak/keycloak:latest`) and `keycloak-postgres` service. Keycloak's DB is deliberately separate from the application `movielens` DB — Keycloak's schema is opaque, backed up on a different cadence, and should not share a lifecycle with application data.
- **New source tree.** `src/auth/` gets the middleware, JWKS cache, and issuer-to-tenant resolver. `synthetic/auth/` gets the token-minting helper for CI and the synthetic-user harness. `infra/keycloak/realms/` holds the seed realm JSONs, tracked in git.
- **FastAPI middleware.** Every route except `/healthz` is wrapped by a dependency that (i) reads the `Authorization: Bearer <token>` header, (ii) verifies signature against the issuer's JWKS, (iii) checks `exp` and `nbf`, (iv) resolves the issuer URL to a tenant slug, and (v) attaches `(tenant_id, user_id)` to `request.state`. Downstream code never sees a raw token or a request without a resolved tenant.
- **CI.** The GitHub Actions serving job brings up Keycloak alongside Postgres and MLflow, seeds a `dev` realm, and mints tokens for the synthetic-load smoke test. Adds ~15 seconds to job setup — within the budget non-negotiable #11 leaves.
- **Frontend.** Next.js integrates via `authjs` (or NextAuth) with the Keycloak provider. The Phase 3 baseline frontend gets a login button; the demo-impersonation mode from CLAUDE.md's frontend scope remains dev-only, gated behind an explicit build flag.
- **Reproducibility.** Realm state lives in `infra/keycloak/realms/*.json` (exported via `kc.sh export`) and is imported at container startup. Development users are seeded programmatically; production users would live in a separate migration path (out of scope for this ADR).
- **Deferred to future ADRs / PRs.** Multi-tenancy isolation mechanism (next ADR, gated by this one). Audit-log schema (bundled with the Phase 3 audit-log code PR). MFA, LDAP federation, SAML (all supported by Keycloak, none enabled). Key-rotation cadence and monitoring (Keycloak rotates keys, but the JWKS cache TTL and rotation-detection alerts are Phase 5 monitoring work).

## Risks

- **JVM startup adds CI flakiness.** Keycloak's cold start is 8–15 s on healthy hardware; a slow CI runner or a container-cache miss can push it over the health-check timeout. Mitigation: the compose health check waits up to 60 seconds with 2-second retries; the CI job uses `depends_on: keycloak: service_healthy` rather than a fixed sleep. If flakiness materializes, the escape hatch is Keycloak's optimized native image (`kc.sh build --optimized`) with sub-second startup at the cost of a heavier CI build.
- **Realm state drift between dev/staging/prod.** If the seeded realm JSON in `infra/keycloak/realms/` falls out of sync with the live realm (a user is added via admin UI in dev), the "reproducibility test" non-negotiable is silently broken for the auth layer. Mitigation: the realm-export step is a Makefile target (`make keycloak-export-realms`), and a CI check that the exported realm JSON matches the seeded one fails the build if they diverge.
- **Cross-tenant token acceptance via misconfigured issuer trust.** The middleware trusts a set of issuer URLs; if the set is wrong (e.g., trusts all realms on a given host without checking the realm slug matches the target endpoint's tenant), a token from realm A could be accepted at an endpoint that should only accept realm B. Mitigation: the middleware derives tenant from the token's issuer and *does not accept a mismatched `tenant_id` from any other source*. The tenant-isolation integration test in CI (per non-negotiable #9) authenticates as tenant A and asserts no tenant B endpoint returns data.
- **JWKS cache staleness after key rotation.** Keycloak rotates keys; if the middleware's JWKS cache TTL is too long, tokens signed with the new key are rejected until the cache expires. Mitigation: TTL of 5 minutes, and the cache also refreshes on a signature-verification failure before rejecting the token (one retry per request). The retry path is logged so a persistent rotation failure is visible.
- **Dev-mode bypass reaches production.** `dev_auth_bypass=True` in a production compose file would bypass all auth. Mitigation: `Settings` raises at import time if `dev_auth_bypass` is truthy and `environment != "dev"`. The production Docker image sets `ENVIRONMENT=production` in its base layer so the assertion fires before FastAPI can accept a request.
- **Keycloak backup/restore is real ops work we're deferring.** Realm config is exportable to JSON and lives in git — recoverable. User data (created in dev/CI) is ephemeral. Production user data would need Postgres backup of the Keycloak DB, which is a Phase 4+ operational concern this ADR does not solve.

## How we'd know we're wrong

- **CI synthetic-load smoke test's setup time exceeds 60 seconds.** Would mean Keycloak's startup is heavier in the CI runner than assumed. Escape hatch: the optimized native image (sub-second startup) is one build-step change away. If even the optimized image is too heavy, the choice moves to a token-mint script that pre-mints tokens against an ephemeral Keycloak once per suite rather than per test.
- **Realm-per-tenant runs out of headroom.** Keycloak scales to hundreds of realms comfortably; thousands is harder. This system's tenant count is bounded by portfolio demonstration needs (single digits, plus a `demo` realm), so headroom is a non-issue at project scale. But if the system ever hosted real customers, the realm-per-tenant decision would want a revisit against attribute-mapped isolation. That review is not blocked by this ADR — the middleware contract (rationale #6) keeps the swap bounded.
- **The next ADR picks per-tenant FastAPI instances.** Realm-per-tenant plus process-per-tenant would be redundant isolation — the process boundary alone is the strongest form. Would suggest realm-per-tenant is over-engineered and attribute-mapped multi-tenancy in a single realm was the right shape. The next ADR's decision is what surfaces this; realm-per-tenant remains fine either way, just less load-bearing than expected.
- **The synthetic-user harness's token-minting helper grows to hide a 200-line Keycloak-idiosyncratic API surface.** Would suggest we're paying for a real IdP but only using the token-issuance sliver — that's the argument for Postgres-JWT, and it would be time to reconsider. A ~30-line helper is fine; a 200-line one is a signal.
- **A bug in the middleware's issuer-to-tenant resolver ships to production.** The tenant-isolation integration test in CI (non-negotiable #9) is what stops this; if that test doesn't catch a resolver bug, the test is under-designed and the ADR's assumption that issuer-derived tenant is safer than claim-derived is weaker than claimed.
