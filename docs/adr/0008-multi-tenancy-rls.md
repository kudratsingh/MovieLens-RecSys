# ADR 0008 — Multi-Tenancy Isolation: Postgres Row-Level Security

**Status:** Accepted
**Date:** 2026-07-03

## Context

[ADR 0007](0007-auth-provider-keycloak.md) pinned realm-per-tenant Keycloak: every authenticated request arrives at FastAPI with a resolved `tenant_id` (derived from the token's issuer URL) attached to `request.state`. This ADR answers the follow-up question: **once we know which tenant a request belongs to, what enforces that the request cannot see any other tenant's data?**

CLAUDE.md non-negotiable #9: *cross-tenant leakage is the highest-severity bug class.* The isolation choice is the single most load-bearing decision in Phase 3 — a wrong answer here does not fail loud in unit tests; it fails silent in production. The three families:

1. **Schema-per-tenant.** Each tenant gets its own Postgres schema; queries run with `SET search_path TO tenant_x, public`. Tables in the wrong schema are literally not visible.
2. **Row-level security (RLS).** One schema, one set of tables, a `tenant_id` column on every tenant-scoped table, and a Postgres RLS policy that filters rows by `current_setting('app.tenant_id')` on every command.
3. **FastAPI-instance-per-tenant.** One process per tenant, each with its own DB URL. Physical process isolation.

The criteria that decide the choice (from CLAUDE.md, plus one this ADR adds):

- **(a) Query complexity** — what does application code have to look like on every read/write?
- **(b) Blast radius of a bug** — if application code forgets to scope, does the DB catch it, and how loudly?
- **(c) Operational overhead** — migrations, backups, adding a tenant, running the stack.
- **(d) Model registry sharing** — MLflow is one service; how does the isolation choice compose with it?
- **(e) Defense-in-depth** — is a bug in application code caught by an independent layer (the DB), or does the DB trust the app?

## Decision

Multi-tenancy is enforced by **Postgres row-level security** on every tenant-scoped table, with the tenant identifier propagated from the middleware to the DB session via a request-scoped `SET LOCAL`.

Concretely:

- **Every tenant-scoped table declares `tenant_id TEXT NOT NULL`** with a foreign key to `public.tenants(id)`. The `tenant_id` value is the Keycloak realm slug (ADR 0007's identifier — one identifier for the tenant across the whole system).
- **RLS is enabled and forced on every tenant-scoped table.** The policy for `SELECT`, `INSERT`, `UPDATE`, and `DELETE` is `USING (tenant_id = current_setting('app.tenant_id', true)) WITH CHECK (tenant_id = current_setting('app.tenant_id', true))`. `FORCE ROW LEVEL SECURITY` is set so table owners are also subject to the policy.
- **Cross-tenant tables live in `public` without RLS.** `public.tenants` (tenant registry), `public.audit_log` (per-request audit — has a `tenant_id` column but no RLS because the auditor needs cross-tenant visibility), and MLflow's registry tables (managed by MLflow, out of scope for our RLS setup).
- **Two DB roles.** `app_user` — the role FastAPI connects as; non-superuser; RLS applies. `admin_user` — the role Prefect DAGs and MLflow use; can bypass RLS (`BYPASSRLS`) for cross-tenant analytics and materialization. FastAPI is asserted at startup to be connected as `app_user`, not `admin_user`.
- **Middleware sets the session variable inside a transaction that wraps the request.** The FastAPI middleware (introduced by the auth-implementation PR that consumes ADR 0007) opens a transaction, executes `SET LOCAL app.tenant_id = '<tenant_id>'`, dispatches the request, then commits or rolls back. `SET LOCAL` scopes the setting to the transaction only, so the value cannot leak between requests even if the connection is reused.
- **pgBouncer is configured in transaction pooling mode**, not session mode. Session-mode pooling would preserve `SET LOCAL` across transactions on the same connection — that would be a cross-request leak vector.
- **Feature-store keys in Redis remain tenant-prefixed** (`tenant:<id>:...`) as a separate isolation layer. Redis is not covered by Postgres RLS; the middleware layer that constructs Redis keys handles that scoping independently.

## Rationale

1. **Belt-and-suspenders — the DB catches the bug, and the tenant-isolation test catches the DB catching the bug.** RLS makes Postgres the enforcer of last resort: a bug in application code that forgets to scope by tenant does not leak, because the RLS policy runs on every row fetch and rejects everything not matching the session's `app.tenant_id`. The tenant-isolation integration test (non-negotiable #9) then verifies that this enforcement is actually in place — authenticate as tenant A, hit every endpoint, assert no tenant B data appears in any response. Two independent layers cover the same failure mode, and each catches bugs the other cannot: RLS catches missing filters; the integration test catches missing RLS policies.

2. **Migrations stay simple, and simple migrations are safe migrations.** Non-negotiable #5 (reproducibility) puts weight on `make train` producing the same artifact hash from a fixed seed and dataset version. That property extends to the data layer — a migration that runs once, atomically, on one schema is dramatically easier to reason about than one that loops over N schemas and can partially fail. Schema-per-tenant's "loop over schemas" migration story is not disqualifying at single-digit tenant scale, but it is strictly more failure surface with no offsetting property RLS lacks. Adding a tenant under RLS is `INSERT INTO tenants` plus provisioning the Keycloak realm; adding one under schema-per-tenant is running a migration script inside a new schema, with rollback semantics that fork if any prior schema is already at the new version and any later schema is not.

3. **Model registry, audit log, and cross-tenant analytics remain first-class operations.** MLflow's model registry is inherently cross-tenant — one service holds every tenant's champion, and the Phase 3 tenant router asks "what is the champion model version for tenant X?" as a per-request lookup. Under RLS, the `public.tenants` table and MLflow-managed tables sit outside any RLS policy, so cross-tenant admin operations (per-tenant metric dashboards, drift analyses across the aggregate signal) are ordinary SQL run as `admin_user`. Under schema-per-tenant, the same operations become cross-schema `JOIN`s or `UNION ALL`s across N schemas — more code, more places for the schema list to fall out of sync with reality.

4. **The realm-per-tenant / schema-per-tenant symmetry cost is real but not load-bearing.** ADR 0007 chose realm-per-tenant because it gave the tenant identifier a forge-resistant home (the issuer URL of the signing key). One might argue that schema-per-tenant continues the symmetry — one realm, one schema — and this ADR is breaking a clean 1:1. But the symmetry ADR 0007 needed was *identifier* symmetry, not *isolation-mechanism* symmetry: the realm slug is copied to `public.tenants.id` and used as the RLS session variable, so the same identifier flows all the way through. The isolation mechanism is a separable decision.

5. **RLS's failure mode is severe but *loud enough* when the app is designed for it.** The dangerous failure mode of RLS is a silent zero-row result when `app.tenant_id` is not set — the query succeeds with no error but returns nothing. Two design choices convert this from a silent bug into a caught bug: (i) the middleware wraps *every* request in a transaction that sets `app.tenant_id` before any query runs, so "forgot to set" cannot happen for a single request — it can only happen for a systemic middleware bug that would fail the tenant-isolation test on the next CI run; and (ii) `current_setting('app.tenant_id', true)` returns NULL when unset, and the RLS policy `tenant_id = NULL` evaluates to `NULL` (not `TRUE`) — nothing is visible, but the app also cannot *write* under a NULL tenant (`WITH CHECK` fails), so a partial state can't be created. The failure is silent on read but loud on write; combined with the integration test, this is a manageable risk.

6. **Scale-forward without over-engineering today.** Single-digit tenants is the near-term reality, but the choice made here is the choice we live with. RLS's ceiling is essentially unlimited at any tenant count Postgres can hold connections for. Schema-per-tenant's ceiling is real (thousands of schemas degrade Postgres's `pg_catalog` performance, and the per-schema migration loop becomes ops-heavy). Choosing RLS at ten tenants gives up nothing at that scale and preserves headroom that schema-per-tenant would spend before the project's second phase of growth.

## Alternatives considered

- **Schema-per-tenant.** The symmetric choice with ADR 0007's realm-per-tenant, and the most "belt-and-suspenders" option — tables in the wrong schema are literally not in the current `search_path`, so a bug in the WHERE clause has nothing to leak. Fails loud (`ERROR: relation "tenant_x.ratings" does not exist`) when misused. Rejected because (a) the migration story is genuinely worse (loop-with-partial-failure vs. single atomic migration), (b) cross-tenant admin operations require schema-qualified JOINs that are painful to maintain, (c) the scaling ceiling is lower with no offsetting property at this project's scale, and (d) RLS's failure mode is mitigated by the transaction-scoped middleware pattern in rationale #5. Reasonable people could pick this instead; the tenant-isolation integration test would validate either choice.

- **FastAPI-instance-per-tenant.** Physical process isolation — one FastAPI process per tenant, one DB URL per tenant, no shared memory. The strongest form of isolation possible; nothing can leak because nothing is shared. Rejected on operational cost. At single-digit tenants this means running 5–10 FastAPI processes for ~100 req/hr each, with N copies of the ranker model in RAM, N Redis pools, N pool connection sets. The candidate/ranker architecture *wants* fungibility — any process can serve any tenant's request against the tenant's champion model — and process-per-tenant destroys that. Would be the right choice for a very-high-value-per-tenant system (medical, financial); wrong for this one.

- **Application-only filtering (no DB-enforced isolation).** Every query includes `WHERE tenant_id = %s`. Fastest to build, no DB configuration. Rejected outright — non-negotiable #9 wants DB-layer enforcement, and this option offers none. A bug in application code that forgets the WHERE clause silently leaks; there is no second layer to catch it. This is the option the ADR was written to argue *against*, and it doesn't clear the highest-severity-bug-class bar.

- **Hybrid: schema-per-tenant for hot data, RLS (or nothing) for shared tables.** Considered as a middle ground. Rejected because it multiplies the number of isolation-mechanism decisions a reviewer has to hold — every new table becomes a question of "does this get its own schema or a `tenant_id` column?" — with no clear rule to answer it. One isolation mechanism, applied uniformly, is easier to reason about than two mechanisms applied selectively.

- **Table-per-tenant naming (`ratings_tenant_a`, `ratings_tenant_b`).** A flat variant of schema-per-tenant. Rejected because it inherits schema-per-tenant's migration story without inheriting its `search_path`-based ergonomics — the application would have to construct table names by string substitution, which is the worst of both worlds.

## Consequences

- **Table shape.** Every tenant-scoped table (`ratings`, `movies` if per-tenant catalogs are added, `features`, `predictions`, `audit_log` shares a `tenant_id` column but is not RLS-scoped) declares `tenant_id TEXT NOT NULL REFERENCES public.tenants(id)` and has RLS enabled + forced.
- **DB roles.** Two roles: `app_user` (RLS applies, connects from FastAPI) and `admin_user` (`BYPASSRLS`, connects from Prefect DAGs and one-off admin scripts). Docker Compose wires the FastAPI service to `app_user`; Prefect runs as `admin_user`. `alembic` / migration tool runs as a third role (`migrator`) that owns the tables — RLS still applies unless the migrator explicitly bypasses.
- **Middleware contract.** The auth middleware (from the ADR 0007 implementation PR) is extended to open a transaction, run `SET LOCAL app.tenant_id = '<tenant_id>'`, dispatch, then commit/rollback. Explicit exception handlers must roll back the transaction, or the setting could theoretically persist on a returned connection — pgBouncer transaction mode makes this safe regardless.
- **Connection pool.** pgBouncer in `pool_mode = transaction`. Session mode is banned. A startup assertion queries the pooler mode and refuses to boot if it's wrong.
- **CI: tenant-isolation integration test.** New in `tests/tenant_isolation/`. Seeds two synthetic tenants, authenticates as tenant A via Keycloak (ADR 0007's admin API), hits every endpoint (recommendations, features, audit log, model metadata), and asserts no response payload contains any tenant B row. Runs on every PR touching `src/serving/`, `src/auth/`, or migrations.
- **Prefect / MLflow.** Prefect DAGs run as `admin_user` — they compute features across all tenants and write them tenant-scoped. MLflow's tables live in `public` and are unaffected by RLS; MLflow's own tenant-tagging (per-tenant champion versions) lives in run tags and registry model-version tags.
- **Backup/restore.** `pg_dump` of the whole DB captures everything; per-tenant restore is a `WHERE tenant_id = X` clause on the restore path, not a schema-scoped dump. Simpler for full-system DR, slightly more code for per-tenant point-in-time recovery — the trade the RLS choice makes on ops.
- **Deferred to future ADRs / PRs.** Audit-log schema (bundled with the Phase 3 audit-log code PR). Feast + Redis tenant-key namespacing (ADR 0009 pending — the feature-store choice ADR). Rate-limiting per tenant (Phase 3 code, no ADR needed). Per-tenant Postgres statistics / vacuum tuning (Phase 4+ ops concern).

## Risks

- **Middleware fails to run `SET LOCAL` on a code path → silent zero-row result.** The primary RLS failure mode. Mitigation: middleware is one path, exercised by every request in the tenant-isolation integration test; a "no tenant configured" response asserts an *error*, not empty content, if `current_setting('app.tenant_id', true)` returns NULL when the app expected it set. Additional guardrail: the middleware sets a request-scoped log field `tenant_id_set=true` before any query runs; structured-log audit shows any request where this is missing.
- **FastAPI connects as `admin_user` by misconfiguration → RLS bypassed silently.** Highest-impact configuration mistake. Mitigation: startup assertion queries `SELECT rolbypassrls FROM pg_roles WHERE rolname = current_user` and refuses to boot if the connected role has `BYPASSRLS`. Separate DB URLs in Docker Compose for `app_user` vs `admin_user`; the FastAPI service only ever gets the `app_user` URL.
- **pgBouncer in session mode by misconfiguration → `SET LOCAL` leaks across transactions.** Would break request-scope isolation entirely. Mitigation: startup assertion checks the pooler's `SHOW POOLS` output for `mode = transaction` and refuses to boot otherwise. pgBouncer config is checked into `infra/pgbouncer/` in git.
- **`SECURITY DEFINER` functions bypass RLS.** Postgres RLS is bypassed inside functions declared `SECURITY DEFINER` when the definer is a superuser or `BYPASSRLS` role. Mitigation: no `SECURITY DEFINER` functions on tenant-scoped tables. Code review catches this; a CI check greps migrations for `SECURITY DEFINER` and requires an explanatory comment when it appears.
- **Batch/backfill jobs need `admin_user` and must pass `tenant_id` per row.** Prefect DAGs that ingest raw MovieLens data, materialize features, or run per-tenant model training all connect as `admin_user` and bypass RLS. They must set `tenant_id` explicitly on every write. Mitigation: the ingest / materialization DAGs have their own unit tests that assert every insert carries a non-NULL tenant_id; missing-tenant-id would fail the FK constraint on `public.tenants(id)` anyway, but the tests catch it earlier and with a clearer message.
- **Cross-tenant analytics queries accidentally exposed via the API.** An endpoint that returns aggregate metrics (e.g. "count of predictions in the last hour") could, if implemented with `admin_user` credentials, leak per-tenant counts to any authenticated caller. Mitigation: no API endpoint runs as `admin_user`. Any aggregate metric shown in the frontend is either (a) scoped to the caller's tenant (RLS enforces), or (b) explicitly aggregated across tenants in a way that reveals no per-tenant data (e.g. "system-wide QPS" is fine; "QPS by tenant" is not).

## How we'd know we're wrong

- **The tenant-isolation integration test passes on a real leak in staging.** Would mean the test's coverage of endpoints or its assertion strictness is under-designed. Fix by adding negative cases (queries expected to return zero rows must return zero *and* the middleware must have logged `tenant_id_set=true`), plus a chaos-injection test that removes the `SET LOCAL` on a code path and asserts the integration test fails.
- **We accumulate `SECURITY DEFINER` functions to work around RLS for admin operations.** Would suggest the two-role split (`app_user` / `admin_user`) is too coarse — perhaps there's a third role for read-only cross-tenant admin queries, or the API needs a proper admin-scoped endpoint surface instead of application-level workarounds. This ADR does not solve for that shape.
- **Postgres query plans degrade under RLS on hot tables.** RLS adds a filter to every query plan, which Postgres usually optimizes as an index scan on `tenant_id` (a hash index on `(tenant_id, other_key)` compound indexes work well). If we observe query-plan blowups — join reorderings that ignore the `tenant_id` filter, unexpectedly slow scans — it would mean the composite index strategy needs work, or specific hot tables should have `tenant_id` as the leading column of their primary key. Not evidence against the ADR; evidence for a Phase 4 performance-tuning task.
- **Tenant count grows into the hundreds and per-tenant reporting becomes the primary read pattern.** RLS handles the read scale fine, but if the *reporting* layer becomes the bottleneck (aggregate cross-tenant queries dominate CPU), the argument for isolating hot data per-schema strengthens. This ADR does not preclude adding schema-per-tenant *for a specific hot table* as a targeted future change; the base architecture stays RLS.
- **We ship a bug where a request's `tenant_id` is `NULL` and the API returns 200 with an empty payload.** Would mean the "no tenant configured" case is not being caught before the DB query — the middleware should refuse to dispatch a request that reaches its query stage without a resolved tenant. Fix by promoting the missing-tenant case to an explicit `500` (or `403`) before it can reach a DB call.
