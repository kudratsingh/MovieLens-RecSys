# ADR 0005 — Ranker Choice: LightGBM Over a Neural Ranker

**Status:** Accepted
**Date:** 2026-07-02

## Context

[ADR 0003](0003-two-stage-architecture.md) pinned a two-stage architecture and named LightGBM as the ranker, but only in passing — it noted that "LightGBM over a neural ranker" deserves its own ADR. [CLAUDE.md](../../CLAUDE.md) lists this as one of the locked-in stack choices ("Models: PyTorch (two-tower), LightGBM (ranker)") and queued this ADR as the last Phase 2 code decision. This document writes down *why* LightGBM is the ranker before the ranker code lands, so the alternatives get rejected on paper rather than retrofitted as justification later.

The decision being pinned is **the ranker architecture family** and **the training-data construction rule**, not the specific hyperparameters (`num_leaves`, learning rate, regularization, etc.) — those get tuned with the model and live in the run config.

## Decision

**LightGBM** is the ranker. Specifically: a gradient-boosted decision tree (GBDT) model trained on per-(user, item) tabular features pulled from the feature module ([`src/features/`](../../src/features/)), scoring the ~500 surviving candidates from the candidate stage and returning the top-K by predicted relevance.

Training mode is **LambdaRank** (LightGBM's listwise learning-to-rank objective, `objective="lambdarank"`), aligned with [ADR 0001](0001-evaluation-protocol.md)'s NDCG@10 evaluation metric — train the ranker on the loss that approximates the metric we score it on.

**Training-data construction (pinned here, not in the run config):**

- **Positives** are (user, item, timestamp) interactions drawn from train. Each interaction becomes one positive example, with features computed from the user's history *strictly before* that timestamp — same point-in-time invariant [ADR 0006](0006-two-tower-retrieval-architecture.md) pinned for the two-tower's history encoder, enforced by the same class of canary test.
- **Negatives** are sampled from the candidate model's top-K output for that user (excluding the positive itself). This teaches the ranker to distinguish "items the candidate stage surfaced that the user *did* consume" from "items the candidate stage surfaced that the user *didn't*" — the exact discrimination the ranker will make at serving time. Random-negative sampling (i.i.d. from the catalog) is rejected as a variant because it teaches an easier problem than the one the ranker faces in production.
- **LambdaRank groups** are per (user, timestamp) query — each group is `[positive, neg₁, ..., neg_K]`. Group sizes are passed to LightGBM via the `group` parameter.
- **Feature vectors** are computed once per (user, item, as-of-timestamp) triple by `src/features/build_features(...)`, which honors the point-in-time rule and is the code path the tests bind to.
- **Training-set subsampling** is a config knob — the full 20 M train interactions are more than the ranker needs to converge, and the initial config caps the training set at a fixed sample so an iteration cycle stays minutes, not hours.

The ranker code lands in `src/models/ranker/lgbm.py` under a `LGBMRanker` class exposing `fit(features_df, group_sizes, labels)`, `predict(features_df) -> scores`, and `rank_candidates(candidates_by_user, features_by_user) -> dict[user_id, list[movie_id]]` — the last is the shape the training script's evaluation loop and Phase 3's serving handler both want.

Ships with `src/training/ranker.py` (the load → candidates → features → fit → predict → evaluate → log skeleton every Phase 2 model uses), `tests/unit/test_lgbm.py` and `tests/unit/test_features.py`, a `Makefile` `train-ranker` target, and a `CLAUDE.md` Phase 2 status bump marking the phase code-complete.

## Rationale

1. **Tabular features are GBDT's home turf, and the ranker's features are entirely tabular.** Per [CLAUDE.md](../../CLAUDE.md), the ranker consumes user-history aggregates, popularity windows, genre affinities, and recency features — all numeric, low-cardinality categorical, or already-aggregated. Decades of benchmark evidence (Shwartz-Ziv & Armon 2022 *Tabular Data: Deep Learning Is Not All You Need*; Grinsztajn et al. 2022 *Why do tree-based models still outperform deep learning on typical tabular data?*) shows GBDTs beat or match deep models on tabular tasks at this scale, with a fraction of the engineering effort. Reaching for a neural ranker for tabular features at 62 k items is choosing the harder tool to use it badly.

2. **LightGBM hits the p99 < 100 ms latency SLO with room to spare.** Non-negotiable #4 caps p99 at 100 ms end-to-end. LightGBM's tree-based inference is ~1 µs per (user, item) prediction at typical depths; scoring 500 candidates is ~500 µs of pure inference, plus a few hundred µs of feature payload assembly. That leaves >95 ms of budget for candidate generation, feature-store lookups, and network. A small MLP could match this; a transformer or DLRM-style model could not without aggressive batching that complicates the request path.

3. **Training time is minutes, not hours — feature engineering iteration cycle stays tight.** A LightGBM model on MovieLens-25M holdout-scale data trains in 1–5 minutes on a single CPU. Feature engineering is the highest-leverage activity for the ranker's success (CLAUDE.md non-negotiable #2: "watch for leakage in feature engineering"); the iteration cycle dominates the total time-to-good-ranker. A neural ranker at the same scale would be 1–2 orders of magnitude slower to train, which silently penalizes feature-engineering experimentation. The cheap thing to iterate on is the thing that gets iterated on.

4. **Native feature importance + SHAP gives the Phase 4 explainability surface for free.** [CLAUDE.md](../../CLAUDE.md) commits Phase 4 to a *"why this recommendation?"* panel — top contributing features per (user, item) shown to the user via the Next.js frontend. LightGBM's `feature_importances_` and the `shap` library's `TreeExplainer` produce exact per-prediction attributions in milliseconds. A neural ranker's analogues (Integrated Gradients, attention weights) are either slower, less exact, or both — and they don't survive an API-latency budget. Picking the ranker that makes Phase 4's UX cheap is the right time to make that trade, not after Phase 4 demands it.

5. **CPU-only inference keeps the serving stack simple.** LightGBM serves on the same CPU the FastAPI process already runs on. No GPU dependency, no model-server sidecar (Triton, TorchServe), no ONNX/TensorRT export step, no version-skew between training framework and serving runtime. The Phase 3 serving surface stays a single Python process plus Redis. A neural ranker drags one or more of those complications into the build.

6. **Industry convergence at the two-stage ranker stage.** Production ranking systems — LinkedIn's feed ranker, Yelp's search ranker, Etsy's search ranker, Airbnb's experience-personalization ranker, Microsoft's LightGBM-originated MSLR work — have all shipped GBDT rankers in front of (or instead of) neural rankers for a decade. The convergence is strongest precisely in the two-stage shape we're building, where the ranker only sees 100–1 000 candidates, the features are heavily engineered, and explainability is a product requirement. Neural rankers dominate at *retrieval* scale (the candidate stage's domain) and in cross-feature-rich domains (CTR prediction on ads, where DCN-v2 and DLRM excel); the *ranker-after-retrieval* slot is still GBDT-dominant.

7. **Candidate-model-sampled negatives, not random-catalog negatives.** The training-data construction rule (Decision, above) says negatives are drawn from what the candidate model actually surfaced for that (user, timestamp) query. Random-catalog negatives teach the ranker to distinguish "watched a movie" from "watched none of 62 k random items" — a much easier problem than production. The candidate-model-sampled variant teaches the exact discrimination the ranker faces at serving time (100 % of ranker queries have a candidate-model prefix). This is the ranker analogue of the sampled-softmax choice in ADR 0006 rationale #3 — both are corrections against a mismatch between the training-time negative distribution and the deployed one.

## Alternatives considered

- **DLRM (Deep Learning Recommendation Model).** Meta's flagship, designed for very-high-cardinality categorical features (billions of ad ids, user ids, etc.) with embedded feature-cross structure. Rejected on two counts. First, MovieLens 25M has 62 k items and ~162 k users — DLRM's embedding-table scale advantage is irrelevant here. Second, the cross-feature interactions DLRM is good at (e.g. `user_id × ad_id × context_hour`) are already partially captured by the candidate stage's two-tower in our architecture; piling a second deep cross-feature model on top is redundant. Strong fit for the candidate stage's domain (ADR 0003 already pinned PyTorch + ANN there), wrong fit for the ranker stage.

- **DCN-v2 (Deep & Cross Network v2).** Explicit feature crossing as a layer primitive; strong on CTR-prediction benchmarks. Rejected because (a) its win condition is high-cardinality categorical features, which we mostly don't have at the ranker stage, and (b) GPU-required training + serving raises the bar on the entire infra stack to win a few NDCG points on a portfolio dataset. Possible Phase 6 challenger if a real reason to upgrade materializes — but the burden of proof is on the deep model.

- **Simple MLP ranker.** A 2–3 layer dense network on the same tabular features. Defensible: small enough to serve fast, simple enough to train quickly. Rejected because at this feature shape and scale, GBDTs reliably beat MLPs (per the tabular-deep-learning literature above), and the MLP buys no advantage in return — feature importance is harder to extract, hyperparameter tuning is more finicky, and there's no infrastructure-simplification payoff because we'd still want SHAP-equivalent explainability. The MLP is strictly dominated by LightGBM in this slot.

- **XGBoost or CatBoost.** Sibling GBDT libraries. This is the hardest alternative to dismiss because the modeling outcomes are very close. LightGBM wins on three pragmatic margins: (i) leaf-wise tree growth and histogram-based splits give it the fastest training in this family for the data sizes we'll see; (ii) LightGBM's `lambdarank` objective is well-tested and well-documented for two-stage rankers — XGBoost's `rank:ndcg` is comparable but the docs / examples / community recipes lean LightGBM; (iii) the Python integration is clean and the model is easy to serialize for MLflow registry. CatBoost's edge is high-cardinality categorical handling, which the candidate stage absorbs in our architecture. The tie-break could reasonably go to XGBoost; LightGBM is the default because the project lead has the most LightGBM muscle and the integration surface is the most heavily-trodden.

- **Two-tower with a deeper interaction head instead of a separate ranker.** Some production systems collapse stages by adding a small interaction MLP on top of two-tower embeddings and skipping the dedicated ranker. Rejected because it defeats the purpose of the two-stage split ([ADR 0003](0003-two-stage-architecture.md) rationale #2) — the candidate stage optimizes for recall over the long-tail catalog, the ranker stage optimizes for precision on the survivors, and one model can't be excellent at both. Also rejected because LightGBM on engineered features is half the muscle group Phase 2 is built to teach; collapsing the ranker forfeits that learning.

- **Point-wise binary classification (`objective="binary"`) instead of LambdaRank.** Train a classifier on (user, item, label) tuples where label = 1 if consumed, 0 otherwise; rank at inference by predicted probability. Simpler, well-understood, and honestly competitive on many tabular tasks. Rejected because point-wise loss optimizes calibrated probability, and we don't need calibrated probability — we need the *ordering* of candidates for one user to match the ordering of their true relevance. LambdaRank directly optimizes NDCG (via approximate lambdas), which is the metric ADR 0001 pins. On production data, the gap is a few NDCG points at k=10, which is exactly the range that matters. The point-wise variant would be a defensible fallback if LambdaRank were flaky at this scale, which it isn't.

## Consequences

- **Code layout.** `src/models/ranker/lgbm.py` defines `LGBMRanker` with `fit(features_df, group_sizes, labels)`, `predict(features_df) -> np.ndarray`, and `rank_candidates(candidates_by_user, features_by_user)`. The last method matches the shape both the training script's evaluation loop and Phase 3's serving handler will call — no re-wrapping needed at the stage boundaries.

- **Training pipeline.** `src/training/ranker.py` follows the same skeleton every Phase 2 model uses: load split → run candidate generator (item-item for the first ranker run, then two-tower once its recall lift is confirmed) → sample training pairs → fetch features → fit ranker → predict for holdout → `evaluate()` for NDCG@10 → log to MLflow under a new experiment `phase-2-ranker`.

- **Eval-harness reuse.** [ADR 0001](0001-evaluation-protocol.md) already defined NDCG@K=10; no protocol change. The training script calls `evaluate(...)` with the same signature the candidate models use — the ranker's per-user top-10 output is a `dict[int, list[int]]`, exactly what `evaluate` expects.

- **Feature module.** `src/features/` is a first-class code path the ranker consumes. Point-in-time correctness is the module's contract — any feature used at training time must be reconstructible at serving time from data available at the prediction timestamp. Phase 3 swaps this provisional module for Feast (feature views, materialization jobs, online store).

- **Explainability story is set.** Phase 4's *"why this recommendation?"* panel is implemented via SHAP `TreeExplainer` on the deployed LightGBM model. The FastAPI handler exposes a per-(user, item) attribution endpoint; the frontend renders it. No model swap needed when Phase 4 lands.

- **Cold-start handling at the ranker stage.** The ranker's input is the candidate list; if the candidate stage falls through to popularity for a cold user (per ADR 0001 / ADR 0004 / ADR 0006), the ranker still scores those candidates with whatever features are available. For a brand-new user with no history features, the ranker degenerates to ranking by item-side features (popularity, recency) — which is acceptable and matches how cold-start is handled in real two-stage systems.

- **Candidate-generator choice at ranker training time.** The training script pins the champion candidate model per run via config. First run uses item-item (cheapest to fit, results already logged). When two-tower's recall@500 clears item-item's on the ADR 0004 gate, ranker training re-runs against two-tower's candidates. Both runs live in `phase-2-ranker` for direct comparison.

- **Phase 2 closes with this PR.** After the ranker lands, Phase 2 is code-complete: candidate stage (popularity / CF / item-item / two-tower), ranker stage (LightGBM), per-stage evaluation harness, DVC-versioned data. Phase 3's re-scoped enterprise chunk (auth, multi-tenancy, Feast, synthetic-load) opens next.

- **Deferred to future ADRs in this lineage.** Specific LightGBM hyperparameters (`num_leaves`, `learning_rate`, `min_data_in_leaf`, regularization, early-stopping policy) live in run config and are tuned, not pinned by ADR. Feast (Phase 3) gets its own ADR when Phase 3 begins. The ranker promotion-gate threshold (NDCG@10 delta required to beat the current champion) gets pinned alongside Phase 4's promotion-gate code, not here.

## Risks

- **Point-in-time leakage in the feature module.** The severity-highest failure mode, same class as ADR 0006's history-leakage risk. If any feature reads data with `timestamp >= as_of_timestamp` — for instance, an item-popularity window that inadvertently includes the positive interaction itself — offline NDCG@10 inflates arbitrarily while serving-time metrics tank. Mitigation: `tests/unit/test_features.py` includes a strict canary that computes each feature at time `t` on a hand-built fixture and asserts equality with the same feature computed on the same fixture *truncated* to `< t` — proving the feature is a function of strictly-past data.

- **Group leakage across (user, timestamp) queries.** LambdaRank groups the training data by (user, timestamp) — but if the same user appears in multiple groups at different times, the ranker sees their preferences repeatedly and may overfit to individual users. Mitigation: within-user shuffle of training examples before batching, and a `min_data_in_leaf` floor that prevents leaves from capturing single users. Not fully preventable, but detectable via a train / holdout NDCG gap larger than typical for GBDTs.

- **Negative sampling collapses when the candidate model is too weak.** If the candidate model's top-K rarely contains the true positive (low recall@K), most training queries have positives that aren't in the candidate list, and the ranker learns from a distorted negative distribution (no clear "true positive alternative"). Mitigation: the ADR 0004 promotion gate ensures the candidate model is above a recall floor before the ranker trains against its output; below that floor, item-item stays champion and the ranker's negative distribution stays coherent.

- **Objective mismatch on serving.** LightGBM's `lambdarank` optimizes NDCG on the training group distribution; at serving time, the candidate stage's top-500 may have a different score distribution than the training-time negatives (e.g. FAISS returns some noise items). The ranker's calibration on the serving distribution is not guaranteed. Mitigation: the Phase 3 serving-side smoke test scores a small set of known-good queries and asserts the ranker's output ordering matches the offline eval — a shift there flags a distribution mismatch, not a bug in the ranker itself.

- **Feature-store swap in Phase 3 breaks features.** The provisional `src/features/` module and Phase 3's Feast feature views have to produce byte-identical features for the same key at the same as-of-timestamp, or the offline-trained ranker gets served the wrong inputs. This is exactly what non-negotiable #2 (feature parity test) exists to prevent. Mitigation: the feature parity CI test lands in Phase 3 with Feast, and the current module keeps its function signatures stable so Feast can be a drop-in.

## How we'd know we're wrong

- **The LightGBM ranker doesn't improve NDCG@10 over the raw candidate model's ordering.** The ranker's whole job is to re-order the candidate stage's output; if it can't, either the features are uninformative (fix the feature module), the training signal is polluted (check for leakage), or the candidate model's ordering is already near-optimal (rare on real data, but the eval would catch it).

- **Training and holdout NDCG@10 diverge sharply.** A large train / holdout gap (> ~10 points) means the ranker is memorizing users rather than learning transferable patterns. Would trigger a review of `min_data_in_leaf`, regularization, and whether the feature module is inadvertently encoding user identity.

- **NDCG@10 on the two-tower-candidate variant is worse than on the item-item-candidate variant.** Two-tower's recall@500 should exceed item-item's per ADR 0006's promotion gate. If the ranker still does *worse* on top of the better candidate model, the negatives being surfaced by two-tower differ from item-item's in ways the ranker can't handle — signal for either a candidate-generator-aware training run or a re-examination of what negatives look like across models.

- **Feature importances load onto a single feature.** If a single feature (e.g. `item_popularity_all_time`) accounts for > 60 % of feature importance, either that feature is a proxy for a leakage bug or the ranker is not benefiting from the personalized features. Would be caught by inspecting the MLflow-logged feature importances after each run.

- **The point-in-time canary test starts passing spuriously fast.** As in ADR 0006 — the canary is a strict-equality check against a hand-built expected feature value, not a heuristic. If it becomes a rubber-stamp, either the test data got trivialized or the feature computation was reverted to a leakage-friendly path. Guarded by making the fixture deliberately asymmetric (features computed at `t` should differ from features computed at `t + 1` for the same user, and both should be provably reachable from the raw data).
