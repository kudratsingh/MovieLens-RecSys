# ADR 0009 — Feature Store: Feast (Postgres offline, Redis online)

**Status:** Accepted
**Date:** 2026-07-03

## Context

CLAUDE.md's stack table has locked "Feature store: Feast" from day one. This ADR does not reopen that decision; it defends it. Specifically, it defends *Feast versus the two alternatives a design-review will actually raise* — a custom Postgres-views + thin-loader stack, and a fully hand-rolled Redis loader with no framework — because both are cheaper on paper and both look tempting at this project's feature count.

The forcing constraints:

- **Non-negotiable #2 — feature parity test in CI.** An offline-computed feature must match the same feature served online for the same key. This is the single failure mode responsible for most real-world recsys serving bugs; whichever tool we pick has to make parity easy to prove, not hard.
- **Non-negotiable #4 — p99 < 100ms.** Feature reads at serving time have to be sub-millisecond; whatever we choose has to sit behind Redis, not Postgres, at request time.
- **Non-negotiable #5 — reproducibility.** Training on a fixed seed and dataset version must produce the same model artifact. Feature values as of a given training timestamp must be reconstructable, not "whatever the offline store had that day."
- **[ADR 0008](0008-multi-tenancy-rls.md) — RLS + tenant-scoped Redis prefixes.** Every feature read has to carry a `tenant_id`, and the Redis online-store key must include the tenant prefix. The isolation model is set; the feature store has to plumb through it.
- **[Phase 2 features module](../../src/features/pipeline.py)** already ships as `src/features/` — a `FeatureIndex` with point-in-time correctness (a bisect-based per-user / per-item timestamp index). This ADR is what turns that pre-Feast prototype into a Feast repository in Phase 3.

The three families of choice:

1. **Feast.** Declared feature views in Python, materialization from an offline store (Postgres here) into an online store (Redis here), first-class point-in-time joins for training, a registry of what features exist.
2. **Custom Postgres views + thin Redis loader.** Every feature is a materialized view; a small Python job reads views and writes tenant-prefixed keys to Redis. No framework.
3. **Hand-rolled Redis loader (no offline store, no framework).** Every feature is a Python function that computes and writes to Redis. Point-in-time reconstruction for training is re-implemented per feature.

## Decision

The feature store is **Feast with Postgres as the offline store and Redis as the online store**, with the following shape:

- **Feast repository at `src/features/feast_repo/`.** Feature views, entities, and the `feature_store.yaml` config live here. The existing `src/features/pipeline.py` prototype is retired and its logic moved into Feast feature-view definitions.
- **Two entities: `user` and `item`.** Both carry a compound key `(tenant_id, id)` — `tenant_id` is a Feast entity join key on every feature view. This is the shape ADR 0008 requires: every feature row is tenant-scoped in the offline store, and the online-store key derived by Feast includes `tenant_id`.
- **Feature views mirror the Phase 2 module.** User features (interaction count, days-active, days-since-last, genre-affinity), item features (popularity windows, age), user×item features (genre-affinity fraction) — each is a feature view with a defined TTL and a source SQL query against the Postgres offline store.
- **Online store: Redis, key prefix `tenant:<tenant_id>:feast:*`.** The `tenant:<id>:` prefix from ADR 0008 wraps the Feast-generated keys — the online-store key is composed at Feast's side by making `tenant_id` part of the entity, but the Redis instance's key namespace is further prefixed by our Redis client wrapper so tenant scoping is enforced at *both* the Feast layer and the Redis-client layer.
- **Materialization via Prefect.** Two Prefect tasks: `feast apply` (idempotent, applies feature-view changes to the registry) and `feast materialize-incremental` (writes offline → online for a rolling window). Materialization runs on the same DAG cadence as retraining — the retraining flow triggers materialization first, then trains.
- **Feature parity test.** `tests/feature_parity/test_online_matches_offline.py` picks a (tenant, entity, feature, timestamp) tuple, reads the feature offline via Feast's `get_historical_features`, reads the same key online via `get_online_features`, and asserts the values match. This test runs in CI on every PR touching `src/features/` or the offline-store schema.
- **Training-time features via `get_historical_features`.** The ranker's training data assembly ([`src/training/ranker.py`](../../src/training/ranker.py)) is refactored to consume Feast's point-in-time join API rather than the current `FeatureIndex` — the API guarantees no future information leaks into the training row, which was the entire reason `FeatureIndex` implemented its bisect-based lookup by hand.

## Rationale

1. **Point-in-time joins as a first-class API is the single feature we're paying Feast for.** Every real-world recsys has a story where an offline metric looked great and online it collapsed, and the cause was a feature that silently included future information at training time. Feast's `get_historical_features` takes an `entity_df` of `(entity_key, event_timestamp)` tuples and reconstructs feature values *as of each timestamp* — the join is defined in terms of `event_timestamp` on the feature source and `event_timestamp` on the query, and future-timestamped rows are filtered by construction. This is code we already re-implemented once in `FeatureIndex.build` with the bisect trick, and the payoff of using Feast is that we stop owning that code — the failure mode CLAUDE.md flags on the two-tower ("history must only contain items consumed strictly before the positive's timestamp") is exactly what Feast's API is designed to prevent.

2. **A registry is a real artifact, not a design nicety.** `feast apply` writes the current set of feature views to a registry (SQLite or Postgres). The registry is what an operator queries to answer "what features exist, which entities do they attach to, what's their TTL, what's their source?" — questions that in a hand-rolled system are answered by `grep`. The registry also becomes the schema against which the parity test asserts: the test iterates every feature view in the registry rather than a hardcoded list. This is the artifact CLAUDE.md's frontend `explain this recommendation` panel (Phase 4) will read from to render feature attributions with real feature names, not stringly-typed keys.

3. **The parity test is native to Feast's shape, not a bolt-on.** Feast defines two SDK methods that read the same feature by name — `get_historical_features` (offline path) and `get_online_features` (online path). The parity test is a call to each, followed by an equality assertion. In the custom-Postgres-views alternative, the offline computation is a SQL query and the online value comes from a hand-written loader; the parity test has to know about *both* sides of the shape, and any drift is drift the test author has to notice. Feast makes the parity test almost mechanical, which is the property that gets us to actually run it on every PR (non-negotiable #2).

4. **The multi-tenancy fan-out cost is real but bounded.** Feast does not have first-class multi-tenancy — every tenant does not get its own registry. Instead, `tenant_id` becomes an entity join key on every feature view. Materialization for N tenants is either (a) a single materialize call with the entity_df covering all tenants, or (b) a Prefect fan-out with one materialize task per tenant. At single-digit tenants either shape is fine; (a) is simpler and is the default. If tenant count grows, (b) with per-tenant parallelism is a Prefect DAG shape, not a Feast redesign — the fan-out lives in orchestration, not in the feature store.

5. **The design-review defense.** An SRE reading this codebase sees Feast, opens the `feature_store.yaml`, sees Postgres offline + Redis online, and understands the shape without asking. The custom-Postgres-views alternative is a set of SQL files and a loader — every reviewer has to reconstruct what the loader does. The hand-rolled Redis loader is worse: every feature is a Python function whose contract is "this is a feature" and the point-in-time story is per-function. Feast is the *legible* choice, and legibility is what CLAUDE.md's "hand it to an enterprise SRE" framing weights.

6. **The escape hatch is bounded.** If Feast becomes an active problem — the API churns in a breaking way, the multi-tenancy fan-out becomes untenable, the registry becomes an operational burden — the migration path off Feast is well-defined: keep the Postgres offline-store tables and the Redis key schema, replace the SDK calls with either the custom-views alternative or a Feast fork with the specific changes we need. The feature view definitions become the specification for whatever replaces them. This is what makes Feast the right *starting* choice: it's not permanent unless it earns permanence.

## Alternatives considered

- **Custom Postgres views + thin Redis loader.** The honest alternative — we already have Postgres, we already have Redis, and CLAUDE.md's non-negotiables can be satisfied by SQL views + a materialization script. Rejected on rationale #3 (parity test is a bolt-on instead of a native shape) and #5 (design-review legibility). Also rejected on ergonomics at scale: each feature is a view file *plus* a loader entry *plus* a schema-registry entry somewhere, and the point-in-time reconstruction is per-feature SQL. At 30–50 features by Phase 5, this is real sprawl with no offsetting property. Reasonable for a 5-feature system; wrong for what this becomes.

- **Hand-rolled Redis loader (no offline store, no framework).** The "just write Python" answer. Fastest to the first 10 features. Rejected because it dispenses with the offline-store abstraction entirely — training-time feature reconstruction becomes ad-hoc code per feature, and the parity test has nothing to hook into because the "offline" side of the parity is whatever the training script computes. The non-negotiable that this option most badly fails is #5 (reproducibility): a feature that gets recomputed at training time from live data is not reconstructable at a fixed timestamp, which means training on a fixed seed produces different features on different days. Disqualifying.

- **Tecton or Hopsworks (SaaS feature store).** Considered briefly. Rejected on the same axis as Auth0 in ADR 0007: SaaS breaks the local-first ethos, adds an internet dependency to CI, and adds a bill. Also worse than Auth0 in that Tecton is closed-source, so the escape hatch (rationale #6) is much worse. Feast is what an ex-Tecton engineer built as the open-source alternative — this is arguably the same choice, minus the vendor.

- **Feast without an offline store (compute at materialize time, write to Redis only).** Considered as a lighter Feast configuration. Rejected because it forfeits `get_historical_features`, which is the API we're paying Feast for. Without the offline-store side of the shape, we're using Feast as a thin Redis loader with a registry — the loader-alternative is cheaper and clearer at that point.

- **No feature store at all — features live in the ranker training code.** The status quo before this ADR. Rejected because it's Phase 2 shape, not Phase 3 shape. The `src/features/` prototype was always temporary — CLAUDE.md's Phase 2 status specifically flags it as "Provisional home until Phase 3 introduces Feast." This ADR is the promise being kept.

## Consequences

- **Repo layout.** `src/features/feast_repo/` contains `feature_store.yaml`, entity definitions, feature view definitions, and the source SQL for each feature. The existing `src/features/pipeline.py` is retired; `FeatureIndex` is deleted in the code PR that ships this ADR.
- **Dependencies.** `feast[redis]>=0.40` is already in `pyproject.toml` (declared for Phase 3). No new deps at ADR time; the code PR that consumes this ADR is what activates the dependency.
- **Prefect DAGs.** Two new tasks (`feast_apply`, `feast_materialize_incremental`) added to the retraining flow. `feast_apply` is idempotent and runs on every flow invocation; `feast_materialize_incremental` runs with a rolling window computed from the last materialization timestamp.
- **Ranker training refactor.** [`src/training/ranker.py`](../../src/training/ranker.py) is refactored to call `store.get_historical_features` for the training-set assembly step. The current bisect-based `FeatureIndex.features_for` becomes a Feast call; the point-in-time-correctness canary test in `tests/unit/test_features.py` becomes a Feast integration test.
- **Feature parity test.** New at `tests/feature_parity/test_online_matches_offline.py`. Runs in CI on every PR touching features or the online/offline stores. Enumerates every feature in the registry rather than a hardcoded list.
- **Multi-tenancy.** `tenant_id` becomes an entity join key on every feature view. The Redis online-store key is prefixed by `tenant:<id>:` at the Redis-client layer (belt-and-suspenders — Feast's key composition includes `tenant_id`, and the client wrapper prefixes at the network boundary).
- **MLflow interaction.** Feature view versions get logged as MLflow run tags (`feature_view_versions=user_v3,item_v2,user_item_v1`) so a model registered in MLflow is reproducible against a specific feature-registry state, not just a code commit.
- **Deferred.** Feast's on-demand transforms (compute features from other features at request time — useful when combining user features with request-time context) are Phase 4 or later; the ranker at Phase 3 launch reads pre-computed features only. Feast's push API (streaming feature updates) is out of scope until we have a streaming source, which is not in the CLAUDE.md roadmap.

## Risks

- **Feast API churn between minor versions.** Feast has a track record of breaking changes across 0.x releases (0.30 → 0.35 shifted the entity/feature-view API, 0.35 → 0.40 changed materialization semantics). Mitigation: pin `feast[redis]` to an exact minor version in `pyproject.toml` and update deliberately as a bundled PR (upgrade + code fixes + regression test on the parity test). Do not accept transitive-dep resolution to a newer Feast in CI.
- **The registry becomes a hidden state artifact.** The Feast registry (SQLite file or Postgres row) is state that isn't the source code — a mismatched registry vs. current feature-view code produces confusing behavior. Mitigation: `feast apply` runs on every Prefect flow invocation and on every FastAPI startup; the registry is asserted-equal-to-code at both surfaces. Registry state is treated as derived from code, not as authoritative.
- **Multi-tenancy fan-out at materialize time.** With a single materialize task covering all tenants, a slow tenant makes all tenants slow to materialize. With per-tenant fan-out, we've added N Prefect tasks per feature. Mitigation: start with the single-task shape; instrument materialize duration per tenant in Prometheus; if one tenant dominates, split. Do not preemptively fan out — the fan-out is easy to add and hard to unwind.
- **The parity test is only as strong as its sample coverage.** Testing one `(tenant, entity, feature, timestamp)` per feature view means we catch systemic drift but miss data-dependent bugs. Mitigation: the parity test's entity/timestamp fixture rotates across a few pinned combinations per test run; a stronger property-based version is a Phase 5 addition.
- **Feast's cold-start on FastAPI boot.** Loading the registry and connecting to Redis at FastAPI startup adds latency to the first request. Mitigation: `feast apply` is called at startup, and the registry is warmed before the health check reports ready. Measured; if it exceeds the readiness-timeout, the alternative is a pre-warmed sidecar.
- **Deleting `src/features/pipeline.py` orphans the point-in-time-correctness test.** The Phase 2 test `test_point_in_time_correctness` asserts against `FeatureIndex.features_for`. Once Feast replaces it, the test either (a) migrates to Feast, or (b) becomes dead code. Mitigation: the code PR that consumes this ADR migrates the test to Feast's `get_historical_features` API — the same property (features at `t` use only rows with `timestamp < t`) is directly assertable.

## How we'd know we're wrong

- **The parity test still catches offline/online drift that Feast should have prevented.** Would mean Feast's guarantees are weaker than assumed — perhaps materialization timing races, or the online store diverges under load. Fix depends on root cause; may be a Feast configuration issue rather than an ADR-level problem.
- **Framework overhead dominates the code changes for the first 5 new features after this ADR ships.** The whole rationale for Feast is high-ceiling ergonomics; if adding a feature is more code with Feast than the hand-rolled alternative, the choice is wrong for this scale. Fix by revisiting rationale #5 against the observed feature-change diff sizes.
- **Feast's API churns in a way we can't cleanly migrate through.** A breaking release that requires rewriting every feature view is an escape-hatch cost. Fix by exercising rationale #6 — extract the Postgres offline tables and Redis key schema, replace the SDK.
- **Multi-tenancy fan-out becomes the dominant materialize cost by Phase 5.** Would mean the "single task, all tenants" shape doesn't scale to the tenant count. Fix by fanning out — not by moving off Feast.
- **The registry becomes a source of surprise (a drift between code and registry that isn't caught).** Would mean the "assert registry equals code at every FastAPI startup" mitigation isn't strong enough. Fix by adding a CI check that runs `feast apply --dry-run` and fails on any diff.
